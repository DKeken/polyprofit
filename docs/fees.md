## Формула

```
fee = C × p × feeRate × (p × (1 - p))^exponent
```

- **C** = количество shares
- **p** = цена share ($0.00–$1.00)
- **feeRate** = ставка по категории
- **exponent** = форма кривой (1 = линейная, 0.5 = корень, 2 = квадрат)

**Peak fee** всегда при **p = 0.50**. Чем ближе к 0 или 1, тем меньше fee.

## Как минимизировать fee

### 1. POST-ONLY maker orders = 0% fee (ОСНОВНАЯ СТРАТЕГИЯ)

```rust
let order = clob.limit_order()
    .token_id(token_id)
    .price(price)
    .size(size)
    .side(Side::Buy)
    .post_only(true)    // ← гарантия maker = 0% fee
    .build()
    .await?;
```

Если бы ордер сразу исполнился — отклоняется (не платишь fee). Retry с другой ценой.

### 2. Торговать при крайних ценах (p < 0.20 или p > 0.80)

При p=0.20: effective fee = 1.15% (vs 1.80% при p=0.50).  
При p=0.90: effective fee = 0.65%.

Но: при крайних ценах обычно меньше edge.

### 3. Geopolitics рынки = 0% fee

Если найти рынок типа "Will sanctions affect crypto?" в категории Geopolitics → **полностью бесплатно**.

### 4. Sports рынки = 0.75% fee

Если есть сигнал по sports-related crypto event → вдвое дешевле.

## Maker Rebates Program

Taker fees **перераспределяются** maker'ам:

| Категория                                           | Какой % taker fee идёт maker'ам |
| --------------------------------------------------- | ------------------------------- |
| Crypto                                              | 20%                             |
| Finance                                             | 50%                             |
| Sports, Politics, Tech, Culture, Economics, Weather | 25%                             |

**Это значит:** если мы ставим maker orders и кто-то их заполняет как taker — мы **получаем** часть их fee. Дополнительный доход поверх trade profit.

---

## Расчёт fee в коде

```rust
fn calculate_taker_fee(shares: f64, price: f64, category: Category) -> f64 {
    let (fee_rate, exponent) = match category {
        Category::Crypto      => (0.072, 1.0),
        Category::Sports      => (0.030, 1.0),
        Category::Finance     => (0.040, 1.0),
        Category::Politics    => (0.040, 1.0),
        Category::Tech        => (0.040, 1.0),
        Category::Economics   => (0.030, 0.5),
        Category::Weather     => (0.025, 0.5),
        Category::Culture     => (0.050, 1.0),
        Category::Other       => (0.200, 2.0),
        Category::Mentions    => (0.250, 2.0),
        Category::Geopolitics => (0.000, 1.0), // бесплатно
    };

    shares * price * fee_rate * (price * (1.0 - price)).powf(exponent)
}

// Пример: 100 shares @ $0.50 (crypto)
// = 100 * 0.50 * 0.072 * (0.50 * 0.50)^1
// = 100 * 0.50 * 0.072 * 0.25
// = $0.90 fee
```

## Влияние fee на стратегию

| Edge | Taker P&L (1.8%) | Maker P&L (0%) | Разница |
| ---- | ---------------- | -------------- | ------- |
| 10%  | +$0.04/trade     | +$0.05/trade   | +25%    |
| 15%  | +$0.11/trade     | +$0.13/trade   | +18%    |
| 20%  | +$0.18/trade     | +$0.20/trade   | +11%    |
| 30%  | +$0.32/trade     | +$0.35/trade   | +9%     |

**Вывод:** при малом edge (10-15%) разница между maker и taker = **жизнь или смерть стратегии**. Maker orders — обязательны.

## feeRateBps — обязательное поле при подписи ордера

С января 2026, ордера в fee-enabled рынках **должны** включать `feeRateBps` в signed payload. Без него ордер отклоняется.

### Как получить fee rate

```
GET https://clob.polymarket.com/fee-rate?token_id={token_id}
```

**НИКОГДА не хардкодить fee.** Polymarket может изменить ставку в любой момент.

### Как это работает

1. SDK автоматически запрашивает fee rate при `.build().await`
2. Включает `feeRateBps` в signed payload
3. Если fee rate изменилась после подписи → ордер **отклоняется** → retry

### Точность (precision)

Fee всегда **округляется вверх** до ближайшего цента (6 decimal places для USDC). Это значит что при расчёте expected profit учитывай ceiling, не floor.

## Liquidity Rewards Program (отдельно от Maker Rebates)

> Помимо maker rebates, Polymarket запускает **Liquidity Rewards** для отдельных рынков.

### Механика

1. Каждый рынок с LR имеет дневной пул наград (USDC)
2. Каждую **минуту** оценивается scoring функция:
   ```
   Q_spread = max(0, min_spread - |price_i - mid|)^2
   Q_final = Q_spread × order_size_i / sum(order_size_j)
   ```
3. Score нормализуется по эпохам (24 часа)
4. Награды распределяются **ежедневно в 00:00 UTC**

### Требования

- Ордера должны быть активны **минимум 3.5 секунды** (для March Madness, может варьироваться)
- Scoring квадратичный по спреду — чем ближе к mid, тем **экспоненциально** больше награда
- Конкуренция только с другими makers в том же рынке

### Стратегическое значение

Это **третий источник дохода** (помимо trade profit и maker rebates). Для рынков с LR — ставить tight quotes и держать их активными.
